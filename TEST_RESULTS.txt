BUG FOUND & FIXED
-----------------
Bug: _register_to() filled image borders with BLACK by default when aligning
two images via warpPerspective(). Black borders are larger than the diamond
shape and confused find_diamond_center() into detecting the border edge
instead of the actual diamond.

Effect: After any camera shift, the aim point was detected at the wrong
location (e.g. (743, 795) instead of (450, 600)), producing completely
wrong click corrections.

Fix: Added borderValue=(255, 255, 255) to the warpPerspective() call in
_register_to(). White fill matches the target background and the diamond
is correctly found after alignment.

File changed: Calc.ipynb, cell 3, function _register_to()
  OLD: warped = cv.warpPerspective(img_bgr, H, (w, h))
  NEW: warped = cv.warpPerspective(img_bgr, H, (w, h), borderValue=(255, 255, 255))


OBSERVATION 1 -- NOT A BUG
---------------------------
estimate_px_per_cm_from_grid() returns 80 px/cm on perfectly synthetic
grid images, but correctly returns 40 px/cm on real target photos.

Reason: The synthetic grid is perfectly uniform -- every grid line is
identical. This makes all autocorrelation peaks equal-height (at lag=40,
80, 120...) and argmax() picks a later one. Real photos have natural
texture variation that makes the lag=40 peak slightly dominant.

Conclusion: No fix needed. The function works correctly for real usage.


OBSERVATION 2 -- BEHAVIOR BY DESIGN
--------------------------------------
With max_dist_px=None (the default in process_shot), outlier holes far
from the main group ARE included in the group centre calculation.

The code comment already documents this:
    aim_xy=aim_px,
    max_dist_px=None,   # or 300-400 if you want

If you want to reject holes more than 300px from the aim point, change
the process_shot() call to:
    max_dist_px=300
